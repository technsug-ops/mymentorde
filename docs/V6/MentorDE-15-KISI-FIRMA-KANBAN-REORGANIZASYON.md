### 1.2 Kanban Philosophy

```
KANBAN = Deliver value continuously
└─ NOT Agile sprints (1-2 week iterations)
└─ NOT Waterfall (big releases)
└─ Work flows through board as it's ready
└─ Pull work when capacity exists
└─ Real-time visibility
└─ No synchronous ceremonies

FOR MENTORDEFÇ:
├─ Sales: Real-time (always pulling deals)
├─ Operations: Continuous (always processing students)
├─ Development: Flow-based (features ship when ready)
├─ Support: On-demand (respond to tickets)
└─ Marketing: Batch jobs (content schedules)

NO STANDUPS (KANBAN):
├─ Daily 09:00 standup = NOT kanban
├─ Weekly ceremonies = NOT kanban
├─ Sprint reviews = NOT kanban
└─ Instead: Kanban board is the source of truth

YES TO ASYNC (KANBAN):
├─ Slack updates when status changes
├─ Automated notifications (board movements)
├─ Email summaries (daily/weekly)
├─ Chat when blocking (not standup meetings)
└─ One-on-ones only when needed
```

### 3.2 Kanban Rules (No Standup Version)

```
┌──────────────────────────────────────────────────────┐
│        KANBAN WORKFLOW (Without Daily Standups)      │
├──────────────────────────────────────────────────────┤
│                                                      │
│ RULE 1: VISUALIZE THE WORKFLOW                      │
│ ├─ Board is source of truth (not your brain)        │
│ ├─ Move cards when status changes                   │
│ ├─ Anyone can see what's being worked on            │
│ ├─ Card details = task description                  │
│ ├─ No standing meetings needed                      │
│ └─ Slack: "Card moved" webhook → Slack notification │
│                                                      │
│ RULE 2: LIMIT WORK IN PROGRESS (WIP)               │
│ ├─ Developer can't start new task if limit hit      │
│ ├─ Forces completion before starting new            │
│ ├─ Example: DEV column max 2 cards (2 devs)         │
│ ├─ When one finishes → other can start               │
│ ├─ No bottlenecks (everyone can see blockage)       │
│ └─ WIP limits = self-enforcing (no manager needed)  │
│                                                      │
│ RULE 3: MANAGE FLOW                                 │
│ ├─ Pull work when you have capacity (not assigned)  │
│ ├─ Not "manager assigns tasks"                      │
│ ├─ But "developer picks next highest priority"      │
│ ├─ Backlog is prioritized (Halil + Tech Lead do)    │
│ ├─ Devs pull from top of prioritized backlog        │
│ └─ No bottleneck in assignment                      │
│                                                      │
│ RULE 4: MAKE PROCESS EXPLICIT                       │
│ ├─ Columns = explicit workflow states               │
│ ├─ Card movement = status update                    │
│ ├─ All info on card (design spec, link, test cases)│
│ ├─ Don't need meeting to ask "what's happening?"    │
│ └─ Just read the card!                              │
│                                                      │
│ RULE 5: USE SLACK WEBHOOKS (ASYNC UPDATES)         │
│ ├─ When card moves → Slack notification              │
│ ├─ No need for daily standup                        │
│ ├─ #growth-pipeline channel (auto-updates)          │
│ ├─ #product-dev channel (auto-updates)              │
│ ├─ #ops-journal channel (auto-updates)              │
│ │                                                   │
│ │ Example Slack notification:                       │
│ │ "🎉 Deal closed: Company X (10 students)"        │
│ │ "📦 Feature deployed: Leave balance fix"         │
│ │ "✅ Student completed: Ahmed → University ready" │
│ │                                                   │
│ └─ Team stays informed without meetings             │
│                                                      │
│ RULE 6: CONTINUOUS IMPROVEMENT (No Retros)         │
│ ├─ Monthly: Look at board metrics                   │
│ │  ├─ How many deals closed?                        │
│ │  ├─ How long student takes (INTAKE → COMPLETED)? │
│ │  ├─ Feature delivery time?                        │
│ │  └─ Any bottlenecks in columns?                   │
│ │                                                   │
│ ├─ When bottleneck found:                           │
│ │  ├─ Example: Students stuck in DOCS_PENDING      │
│ │  ├─ Action: Add template, auto-send reminder      │
│ │  ├─ Result: Docs collected faster                 │
│ │  └─ No "retro meeting" needed (async discussion)  │
│ │                                                   │
│ └─ Monthly email: "Here's what we improved"         │
│                                                      │
│ RULE 7: NO DAILY STANDUPS                           │
│ ├─ Instead: Kanban board is the standup            │
│ ├─ If you need to know status → look at board       │
│ ├─ If you're blocked → move card to blocked column  │
│ ├─ If you need help → mention person in Slack       │
│ ├─ One-on-ones: scheduled only when needed          │
│ └─ Weekly all-hands: OPTIONAL (for culture, not ops)│
│                                                      │
│ RULE 8: DEPLOYMENT CADENCE                         │
│ ├─ Feature ready (in READY column) → Deploy today   │
│ ├─ Not "wait for end of sprint"                     │
│ ├─ Small deployments = less risk                    │
│ ├─ Deploy multiple times per week OK                │
│ └─ Monitor: If breaks → quick fix                   │
│                                                      │
│ RULE 9: ESCALATIONS (Blocked Issues)               │
│ ├─ If card blocked → move to BLOCKED column         │
│ ├─ Add comment: "Waiting for X from Y"              │
│ ├─ Ping person in Slack (not in standup)            │
│ ├─ Unblock within 24 hours (priority)               │
│ ├─ If can't unblock → pull person into async chat   │
│ └─ No weekly "escalation meeting"                   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 3.3 Communication Pattern (No Standup)

```
INSTEAD OF DAILY STANDUPS:

❌ OLD (Agile):
├─ 09:00 All-hands standup (30 min)
├─ Everyone reports: "Yesterday done/today doing"
├─ Blocking issues discussed
├─ Lots of context switching
└─ Requires synchronization

✅ NEW (Kanban):
├─ NO daily standup
├─ Kanban board = everyone knows status
├─ Slack webhook notifications (automatic)
├─ Async Slack for quick questions
├─ One-on-ones scheduled when needed
├─ Weekly summary email (what shipped)
└─ Monthly all-hands (culture + strategy, not operations)

COMMUNICATION FLOWS:

1. STATUS UPDATES:
   └─ Kanban board + Slack webhooks (automatic)
   └─ No need to say "I'm working on X"
   └─ Card in DEV column = everyone knows

2. BLOCKERS:
   └─ Move card to BLOCKED column
   └─ Add comment on card
   └─ Ping person in Slack: "@John, stuck on X"
   └─ Not in daily standup meeting

3. QUICK QUESTIONS:
   └─ Slack DM or channel (async)
   └─ "Hey, how do I do Y?" → answered within 2 hours
   └─ Thread in Slack (context preserved)
   └─ No disruption to deep work

4. DECISIONS:
   └─ Halil decides (COO) or Tech Lead decides
   └─ Decision made async in Slack
   └─ Card updated with decision
   └─ No meeting required

5. CELEBRATIONS:
   └─ Slack announcement when milestone hit
   └─ "🎉 Deal closed!" / "🚀 Feature deployed!"
   └─ Team morale boost (async)

6. STRATEGIC:
   └─ Monthly all-hands (optional, 1 hour)
   └─ Cover: This month's wins, next month's goals
   └─ Not operational status (board shows that)
   └─ Can be recorded for async viewing

EXAMPLE DAILY FLOW (No Standup):

08:00 - People start work (whenever they want)
        └─ They check Kanban board
        └─ They check Slack for overnight updates
        └─ They pull a card (or continue existing)

12:00-13:00 - Lunch (staggered, not all same time)

15:00 - Blocking issue found
        └─ "Waiting for API spec from John"
        └─ Move card to BLOCKED
        └─ Ping John in Slack: "@John, need API spec"
        └─ John responds within 30 min
        └─ Move card back to DEV

17:00-18:00 - People leave work (flexible end time)
              └─ Update cards before leaving
              └─ Slack webhook sends daily summary
              └─ Everyone knows what happened

NO MEETINGS REQUIRED!
└─ Just async communication + kanban board
```

## BÖLÜM 6: KANBAN METRICS (Instead of Sprints)

### 6.1 What We Measure

```
INSTEAD OF "Sprint velocity (points per 2 weeks)":

✓ KANBAN METRICS:

1. LEAD TIME (How long from idea → done)
   ├─ Growth: Prospect → Closed deal (target: 7 days)
   ├─ Product: Idea → Deployed (target: 5 days)
   ├─ Student: Intake → Mentor assigned (target: 1 day)
   └─ Measured: Automatically by Trello (card created → moved to DONE)

2. CYCLE TIME (How long card spends in DEV/IN_PROGRESS)
   ├─ Product: Start coding → Ready for testing (target: 2 days)
   ├─ Growth: Start negotiating → Close (target: 5 days)
   └─ Student: Docs pending → Mentor assigned (target: 1 day)

3. THROUGHPUT (Cards per week shipped to DONE)
   ├─ Product: 2-4 cards/week done (2 devs)
   ├─ Growth: 3-5 deals closed/week
   ├─ Students: 10-15 completed/week
   └─ Track: Manually in Slack (weekly count)

4. WIP HEALTH (Are we respecting WIP limits?)
   ├─ Dev column: Never exceeds 2 cards (2 devs)
   ├─ Ops column: Never exceeds 30 students
   ├─ Sales column: Whatever, no limit
   └─ Check: Weekly (part of monthly review)

5. BLOCKED CARDS (Are we getting stuck?)
   ├─ Count: How many cards in BLOCKED column?
   ├─ Duration: How long have they been stuck?
   ├─ Resolution: What's the pattern?
   └─ Action: Monthly improvement (reduce blocks)

6. QUALITY METRICS (From existing monitoring)
   ├─ Bug rate (from production)
   ├─ Uptime (from existing monitoring)
   ├─ Student satisfaction (from surveys)
   ├─ Mentor satisfaction (from feedback)
   └─ NO NEW TOOLS (use what you have)

7. BUSINESS METRICS (What matters most)
   ├─ Revenue (monthly €)
   ├─ Student acquisition (students/month)
   ├─ Retention (students completing)
   ├─ Profit margin (€ / cost)
   └─ NPS (student satisfaction)

MONTHLY REVIEW (Instead of Sprint Retro):
├─ Look at metrics above
├─ What's improved? (celebrate it)
├─ What's regressed? (fix it)
├─ Change process? (kanban rule tweak)
├─ Any bottlenecks? (add capacity or process)
└─ Email summary to team (async, no meeting required)
```
